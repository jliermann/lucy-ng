# Phase 94 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase is an **extension**, not a new design. It populates the
"Tables" tab placeholder docked in by Phase 93 with five read-only, formatted data
tables. It introduces **zero new design system, zero build toolchain, zero CDN**.
Every token below is either lifted verbatim from the shipped
`src/lucy_ng/webview/static/index.html` `<style>` block, or is a minimal,
same-family extension of it (documented as "NEW" where added). The executor MUST
NOT invent colors, fonts, or spacing outside this contract.

## Design System

| Property | Value |
|----------|-------|

## Typography

All four roles are pre-populated verbatim from the shipped `#log-panel` rules in
`index.html` (source of truth: this is the only existing "rich content inside a tab

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 12px | 400 | 1.6 (matches shipped `#log-panel p`, `li`; use for table cell text, narrative bullet items, section body text) |
| Label | 12px | 600 | 1.3 (table `<th>` header cells, per-table caption line — caption itself is 400, see Copywriting; the 600 weight applies to the small "Constraint" / count-row key labels only) |

Font sizes in play this phase: **12, 13, 14, 16px** (4 sizes — matches the shipped
scale exactly, no new size introduced). Font weights in play: **400 and 600 only**
(matches shipped scale exactly, no new weight introduced).

## Copywriting Contract

This is a read-only, no-CTA, no-destructive-action dashboard extension. Empty/waiting
copy follows the exact existing voice pattern already shipped ("Waiting for log
data...", "Waiting for candidates...") — one line per table, in the format
`"Waiting for {data} ..."`.

| Element | Copy |
|---------|------|
| Primary CTA | None — this phase adds no interactive controls (no buttons, no forms); the dashboard remains strictly observational per REQUIREMENTS.md "Out of Scope" |
| Empty state — ¹³C table | `Waiting for ¹³C signal data...` |
| Empty state — HSQC table | `Waiting for HSQC correlation data...` |
| Empty state — HMBC table | `Waiting for HMBC correlation data...` |
| Empty state — COSY table | `Waiting for COSY correlation data...` |
| Empty state — LSD constraint inventory | `Waiting for LSD constraint data...` |
| Empty state body (all 5) | No secondary/next-step line needed — matches existing one-line waiting-state precedent (`#log-panel`, `#structure-waiting`) with a live 3 s poll already implicitly communicating "this will fill in automatically" |
| Error state | None surfaced to the user. Per SC4 / RESEARCH.md Pitfall 1 &amp; 5, the backend collapses every failure mode (missing file, partial write, malformed JSON, malformed LSD block) into the same `{"state":"waiting"}` HTTP 200 payload — there is no distinct "error" UI state to design. Network/fetch-level failures (e.g. server unreachable) are silently logged via `console.warn(...)` and never shown to the user, matching the existing `refreshStatus`/`refreshStructures`/`refreshLog` `.catch()` pattern verbatim — do not add a visible error banner for this phase. |
| Destructive confirmation | Not applicable — no destructive actions exist in this phase (read-only display only) |

### Per-table caption copy (D-05 — format contract)

Each table gets one caption line directly above it, styled as small muted text
(12px / 400 / `#6c757d`, matching shipped `#structure-hint`: `font-size:12px; color:#6c757d; padding:4px 0`).
Format: `{note} — {salient counts}`. Examples (illustrative, not literal — actual
`note`/count values come from the source JSON, verbatim, never re-worded by the
frontend):

- ¹³C: `{note} — {formula}, DBE {dbe}, {solvent}`
- HSQC: `{note} — {count} correlations`
- HMBC: `{note} — {kept_count} kept of {raw_count}`
- COSY: `{note} — {count} correlations`
- LSD constraints: `Iteration {iteration} · {formula} · {timestamp}`

If `note` is absent/empty in the source JSON, render only the counts segment (no
dangling em-dash) — a small formatting-defensiveness note for the executor, not a
new design decision.

## Phase-Specific Extensions

The template's standard sections above cover tokens; this phase also needs concrete
**structural/layout** and **data-formatting** contracts that the planner/executor need
to build against. These are extension-only (D-01 through D-05 from CONTEXT.md, made
checkable):

### Tables-tab panel structure (applies to `data-panel="tables"` in `index.html`)

Replace the current placeholder `<div class="placeholder" data-panel="tables">Tables — coming in Phase 94</div>`
with a container holding 5 stacked `<section>` blocks, each independently rendered/
independently "waiting" (SC4), in this fixed order:

1. `#table-carbon` — "¹³C Signals" (Heading, 14px/600/`#212529`)
2. `#table-hsqc` — "HSQC" (Heading, 14px/600/`#212529`)
3. `#table-hmbc` — "HMBC" (Heading, 14px/600/`#212529`)
4. `#table-cosy` — "COSY" (Heading, 14px/600/`#212529`)
5. `#table-constraints` — "LSD Constraint Inventory" (Display, 16px/600/`#212529` — the
   one section that gets the larger Display-role heading, since it's structurally
   richer than the other four)

Each section = heading → caption (D-05) → body (table, or waiting message). Sections
are separated by `md` (16px) vertical gap; no `<hr>` divider needed between them (the
heading + whitespace is sufficient, consistent with the existing single-scroll
`[data-panel]` container — do not add nested scrollbars per section).

### Per-table "waiting for data" state (SC4 — independent per table)

New class `.table-waiting`, distinct from the existing tab-level `.placeholder` (which
is reserved for whole-tab "not yet built" states from Phase 93 and must not be reused
here — Tables tab IS built, individual data sources may just not be ready yet):

```
.table-waiting {
  font-size: 12px;
  font-style: italic;
  color: #6c757d;
  padding: 12px 0;
}
```

Rendered via `textContent` only (no `innerHTML` — carries forward the Phase 93 D-02 /
93-CONTEXT.md XSS discipline; the existing `test_no_innerhtml_in_source` regression
test scans the whole `webview.js` file, so any violation here breaks that test
file-wide).

### TBL-03 structured multi-section layout (D-01 — exact section order and styling)

Within `#table-constraints`, after the Display-role title and caption, render exactly
three subsections in this order:

1. **"Applied Constraints"** (Heading role, 13px/600/`#495057` — h3-equivalent)
   — a `buildTable(['Type', 'Atom Indices', 'Note'], rows)` table. One row per
   constraint instance: `BOND` rows from `bond_constraints` (e.g. `"1 14"`), `HMBC`
   rows from each batch's `correlations` entries (e.g. `"1 9"`), `COSY-equiv` rows
   from `cosy_equiv_pairs`. `Note` column populated from the matching
   `applied_from_detection` narrative string where one maps to that constraint;
   otherwise empty cell (never a placeholder dash/word — a genuinely empty `<td>` is
   correct here, it means "no narrative mapped to this specific row").
   If the source arrays are all empty, render the table with a single body row
   spanning all 3 columns reading `No applied constraints recorded` (12px/400,
   `#6c757d`) rather than an empty `<tbody>` — an empty-looking table with headers but
   zero rows reads as a bug, not as "none".

2. **"Constraint Summary"** (same Heading role, 13px/600/`#495057`)
   — a compact 2-column `buildTable(['Constraint', 'Value'], rows)` table (Label-role
   12px/600 for the left column reads naturally since `<th>` styling already applies
   weight 600 to header cells only — body cells stay Body-role 12px/400). Fixed row
   order: `MULT` (`mult_count`), `HSQC` (`hsqc_count`), `HMBC total` (`hmbc_total`),
   `ELIM budget` (`elim_budget`), `Ring exclusion` (`ring_exclusion_enabled` →
   render as `Yes`/`No`, never raw `true`/`false`), `DEFF/FEXP status`
   (`deff_fexp.status`, `.get()`-defensive per RESEARCH.md).

3. **"Deferred / Pending"** (same Heading role, 13px/600/`#495057`)
   — rendered as a plain `<ul>`/`<li>` bullet list (reusing the exact shipped
   `#log-panel ul`/`li` CSS: `margin: 4px 0 4px 20px`, 12px/400/1.6) from
   `pending_from_detection`, one narrative string per `<li>`, `textContent` only (no
   inline-markdown parsing needed for this data — it is plain prose, not
   CASE-PROGRESS.md markdown). If the array is empty, render one `<li>` reading
   `Nothing deferred this iteration.`

All three subsections separated by `lg` (24px) vertical gap (more than the 16px
between the 5 top-level table sections, because these are visually nested one level
deeper and need clearer separation from their siblings).

### Compact intensity formatting (D-04 — exact algorithm, executor MUST match)

Applies to every `intensity` cell across HSQC, HMBC, and COSY tables. Pure
client-side display formatting (no backend change — RESEARCH.md Architectural
Responsibility Map assigns this to Browser/Client tier):

```
if abs(value) >= 1_000_000: format as (value / 1_000_000) with 1 decimal + "M"   → 5559614 → "5.6M"
elif abs(value) >= 1_000:    format as (value / 1_000) with 1 decimal + "K"       → 5559   → "5.6K"
else:                        render the raw integer unchanged                     → 42     → "42"
```

No thousands separators, no currency-style grouping, no rounding beyond 1 decimal
place. This is a ~5-branch, ~10-line hand-rolled function per RESEARCH.md's
"Don't Hand-Roll" table — no library, no CDN.

### Table density (all 5 tables)

All tables in this phase use the exact shipped `#log-panel table/th/td` density —
do not introduce a denser or looser variant:
- `border-collapse: collapse`, cell padding `4px 8px`, `font-size: 12px`
- `<th>`: `background: #f8f9fa; font-weight: 600; color: #495057; text-align: left`
- `<td>`: `color: #212529`, border `1px solid #dee2e6`
- No zebra-striping (matches existing precedent — the shipped log-panel tables have
  none; do not add alternating row backgrounds, since HMBC's flag-color-coding already
  uses row background as a signal and zebra-striping would visually compete with it)
