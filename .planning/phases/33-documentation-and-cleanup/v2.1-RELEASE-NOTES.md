# v2.1 Working Multi-Agent CASE

**Released:** 2026-02-09
**Milestone:** Multi-agent orchestration for Computer-Assisted Structure Elucidation

---

## Overview

v2.1 delivers the **working multi-agent CASE architecture** that v2.0 designed on paper but never wired up. Where v2.0 provided skill documents and agent definitions without orchestration, v2.1 creates a functioning system: sub-command skills following the GSD pattern, real agent spawning via the Task tool, progress monitoring through structured files, loop detection across 4 patterns, advisory interventions that preserve agent autonomy, and diagnostic specialist delegation after failed recovery attempts.

The system now runs autonomously from NMR data to candidate structures, detecting and recovering from unproductive loops without infinite retries.

## Key Features Delivered

### 1. Sub-Command Skills (Phases 27, 31)

Five focused GSD-pattern skills replacing the monolithic /lucy-ng command:

| Command | Purpose | Key Behavior |
|---------|---------|--------------|
| `/lucy-ng:status` | Environment check | Verifies lucy-ng version, LSD solver, database |
| `/lucy-ng:dereplicate` | 13C matching | Thin wrapper around `lucy dereplicate c13` CLI |
| `/lucy-ng:predict` | Shift prediction | Thin wrapper around `lucy predict c13` CLI |
| `/lucy-ng:sanitise` | Dataset cleaning | AI-driven compound identifier removal (no CLI) |
| `/lucy-ng:case` | CASE orchestrator | Spawns agent, monitors, detects loops, intervenes |

**Pattern:** YAML frontmatter + XML-tagged process steps, allowed-tools restriction, thin CLI wrappers (no domain intelligence in skills).

### 2. CASE Orchestrator (Phase 29)

The `/lucy-ng:case` orchestrator (622 lines) implements supervision logic dissolved from v2.0's supervisor.md:

**12-step orchestration process:**
1. Parse arguments (dataset path)
2. Validate prerequisites (LSD, database)
3. Check dereplication-first policy (never dereplicate in CASE)
4. Spawn CASE agent via Task()
5. Monitor progress through CASE-PROGRESS.md (append-only structured file)
6. Detect loops across 4 patterns
7. Perform basic diagnostics (sp2 count, H count, 1J check)
8. Generate advisory interventions (WHAT not HOW)
9. Track per-pattern intervention counters
10. Escalate after 10 failed cycles per pattern
11. Delegate to diagnostic specialist after 2 failed basic interventions
12. Return final result or escalation

**Loop detection patterns:**
- Threshold cycling: Same thresholds tried repeatedly
- Failed LSD runs: LSD errors without changing constraints
- No solutions: Repeated "no solutions" without changing input
- Stuck on same error: Same diagnostic message across iterations

**Monitoring approach:** Batch monitoring (~10 iterations) vs synchronous, agent runs autonomously between checks.

### 3. Autonomous CASE Agent (Phase 28)

Created `~/.claude/agents/lucy-case-agent.md` (1,148 lines total, ~528 lines critical inlined knowledge):

**Hybrid context inlining strategy:**
- Critical content inlined directly in agent definition (NMR background, peak picking thresholds, LSD syntax, HSQC/HMBC strategies)
- Detailed references via file paths (full diagnostic procedures, confidence scoring tables)
- Guarantees agent has core knowledge in context when spawned, no dependency on runtime file reads

**Key capabilities:**
- Writes CASE-PROGRESS.md in structured format for orchestrator monitoring
- Applies DEPT-guided HSQC picking with adaptive thresholds
- Uses HMBC cross-validation strategies (HMBC-to-C, symmetric-H-to-C)
- Detects NMR assignment errors (missing sp2, H count mismatches)
- Iterates autonomously for ~10 cycles before orchestrator checks progress

**Agent definition structure:**
- XML-tagged system prompt sections (<role>, <context>, <critical_knowledge>, <workflow>, <monitoring_requirements>)
- Inlined critical knowledge tables (thresholds, LSD commands, correlation strategies)
- File references for detailed procedures (`@skill/diagnostic/SKILL.md`)

### 4. Diagnostic Specialist Integration (Phase 30)

Updated `~/.claude/agents/lucy-diagnostic.md` (formerly diagnostic-specialist.md) with hybrid inlining:

**Spawning trigger:** After 2 failed basic interventions from orchestrator
**Inlined knowledge:** Critical LSD manual sections, systematic check procedures
**Delegation handoff:** Orchestrator provides CASE-PROGRESS.md, current state, failed interventions; specialist performs deep diagnosis and returns root cause analysis

**Old vs new:**
- v2.0: diagnostic-specialist.md (paper-only, never spawned)
- v2.1: lucy-diagnostic.md (working delegation, hybrid inlining, ~600 lines critical content)

### 5. AI-Driven Sanitisation (Phase 31)

The `/lucy-ng:sanitise` sub-command implements compound identifier removal without a CLI tool:

**Why AI-only:**
- Compound identifiers appear in varied formats (filenames, paths, title files, pdata dirs)
- Requires reasoning to distinguish compound names from experimental metadata
- No regex or simple rules can handle all cases

**Process:**
1. Claude reads dataset directory structure
2. Identifies all files/directories containing compound identifiers
3. Generates sanitisation plan (renames, [REDACTED] replacements)
4. Executes sanitisation
5. Verifies no identifiers remain

**Use case:** Preparing datasets for blind CASE validation where Claude must not see compound identity.

**Task tool with model: inherit workaround:**
```xml

  <path>~/.claude/agents/lucy-case-agent.md</path>
```

- Prevents context limit issues while guaranteeing core knowledge available

**Applied to:**

- lucy-diagnostic.md: LSD manual critical sections, systematic check procedures

## Key Decisions Made During v2.1

### Orchestration & Supervision

| Decision | Rationale | Impact |
|----------|-----------|--------|
| **Advisory interventions (WHAT not HOW)** | Preserves agent autonomy, avoids directive commands | Orchestrator suggests "check for missing sp2 carbons", not "lower threshold to X" |
| **Per-pattern intervention counters** | Different loop patterns have different root causes | Tracks failures separately for threshold cycling vs LSD errors vs no solutions vs stuck errors |
| **10-cycle escalation threshold** | Balances automated recovery vs preventing infinite loops | After 10 failed interventions for same pattern, escalate to user |
| **Batch monitoring (~10 iterations)** | Efficient vs synchronous per-iteration checking | Agent runs autonomously, orchestrator reads progress after batch |
| **Diagnostic delegation threshold = 2 failed interventions** | Basic checks exhaust quickly, specialist needed for deep diagnosis | After 2 failed basic interventions, spawn diagnostic specialist |

### Architectural Boundaries

| Decision | Rationale | Impact |
|----------|-----------|--------|
| **/lucy-ng:case NEVER dereplicates** | Absolute separation of concerns | Dereplication is a separate sub-command; CASE workflow is blind structure elucidation only |
| **Sanitisation is AI-only (no CLI)** | Compound identifiers require reasoning to detect | No `lucy sanitise` CLI command; /lucy-ng:sanitise skill uses file operations and AI judgment |
| **Supervisor.md dissolved into case.md** | Supervision is orchestration, not a separate agent | v2.0's supervisor agent concept replaced by orchestrator skill with embedded logic |
| **Hybrid context inlining (~500 lines critical)** | Guarantees agent has core knowledge when spawned | No runtime dependency on file reads for critical content; detailed references for deep dives |
| **[REDACTED] for sanitisation** | User must not see compound identity in blind tests | Orchestrator uses literal [REDACTED] string replacement for compound names |
