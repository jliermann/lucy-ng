  <name>Task 1: Extend WelfordAccumulator with neighbour tracking</name>
  <files>
    src/lucy_ng/prediction/stats_generator.py
  </files>
  <action>
1. Add import at top of file: `from lucy_ng.prediction.hose_parser import parse_sphere_1`

2. Add 5 new fields to WelfordAccumulator dataclass (after sp1_count):
   - has_carbon_neighbor: int = 0
   - has_oxygen_neighbor: int = 0
   - has_nitrogen_neighbor: int = 0
   - has_sulfur_neighbor: int = 0
   - has_halogen_neighbor: int = 0

3. Add method `update_with_neighbors(self, value: float, hybridisation: str, neighbors: dict[str, int]) -> None`:
   - Calls self.update_with_hybridisation(value, hybridisation) to handle shift + hybridisation
   - Then checks neighbors dict for element presence:
     - If "C" in neighbors and neighbors["C"] > 0: self.has_carbon_neighbor += 1
     - If "O" in neighbors and neighbors["O"] > 0: self.has_oxygen_neighbor += 1
     - If "N" in neighbors and neighbors["N"] > 0: self.has_nitrogen_neighbor += 1
     - If "S" in neighbors and neighbors["S"] > 0: self.has_sulfur_neighbor += 1
     - For halogens: if any of "F", "Cl", "Br", "I" is in neighbors with count > 0: self.has_halogen_neighbor += 1

4. Update merge() method:
   - Add 5 lines after sp1_count merge: `merged.has_carbon_neighbor = self.has_carbon_neighbor + other.has_carbon_neighbor` (and same for O, N, S, halogen)
   - Also include neighbour fields in the zero-count shortcut returns at the top of merge().

5. Update to_tuple() return:
   - Change from 6-element to 11-element tuple:
     `return (self.count, self.mean, self.m2, self.sp3_count, self.sp2_count, self.sp1_count, self.has_carbon_neighbor, self.has_oxygen_neighbor, self.has_nitrogen_neighbor, self.has_sulfur_neighbor, self.has_halogen_neighbor)`
   - Update docstring to document the 11-element format.

6. Update both ResumableHOSEStatsGenerator._process_chunk() and SDFHOSEStatsGenerator._process_chunk():
   - After generating hose_code, add: `neighbors = parse_sphere_1(hose_code)`
   - Change the accumulator update call from:
     `accumulators[(hose_code, radius)].update_with_hybridisation(shift_ppm, hybridisation)`
     to:
     `accumulators[(hose_code, radius)].update_with_neighbors(shift_ppm, hybridisation, neighbors)`

7. Update HOSEStatsGenerator.generate_all():
   - The return signature changes. Currently returns `(aggregates, hybridisations)`.
   - Add a third return dict: `neighbours: dict[tuple[str, int], dict[str, dict[str, int]]]` which maps (hose_code, radius) to per-observation neighbour dicts.
   - HOWEVER, this class uses list-based aggregation (not Welford), so tracking per-observation neighbours is complex.
   - SIMPLER: Add neighbour counting inline like hybridisation. Create `neighbour_counts: dict[tuple[str, int], dict[str, int]]` with default `{"C": 0, "O": 0, "N": 0, "S": 0, "halogen": 0}`. For each HOSE code generated, parse sphere 1 and increment the relevant element counter if element is present.
   - Return as third element: `return dict(aggregates), dict(hybridisations), dict(neighbour_counts)`
   - Update compute_stats() to accept optional `neighbour_counts` parameter and set the fields on HOSEStatsRecord.
   - Update populate_database() to pass neighbour_counts through.

IMPORTANT: The _upsert_chunk_stats methods in ResumableHOSEStatsGenerator and SDFHOSEStatsGenerator already work correctly because they call `acc.to_tuple()` and prepend (hose_code, radius). Since to_tuple() now returns 11 elements, the resulting tuple is 13 elements, which manager.py's upsert_hose_stats_incremental (updated in 35-01) handles.
  </action>
