## Architecture Patterns

### Pattern: Additive Extension of Existing Sections

All three agent files follow a consistent markdown structure with named sections using `##` and `###` headings. All changes must follow this pattern — add content within existing sections or add a new subsection that clearly belongs conceptually.

**solution-analyst:** Section `## 4. Chemical Plausibility Assessment` has five numbered checks (Check 1 through Check 5). Add Check 6 immediately after Check 5. This is the natural location; the planner should not restructure the section.

**nmr-chemist:** Section with `### [SETUP-COMPLETE]` message template. This template lists fields line by line. Add one new field `Aromatic expectation:` at the end, just before `Key observations:` (which is currently the last field). This ordering is correct because aromatic expectation is a derived observation from the picked spectrum.

**devils-advocate:** Section `## 3. Structural Integrity Checks` lists named checks. Add a new check called "Aromatic Ring Expectation" at the end of this section. Use WARNING severity (not CRITICAL), because: (a) this is defense-in-depth after solution-analyst already flags it, (b) the aromatic warning from ranking is informational/diagnostic rather than a definite error, (c) CRITICAL would block the solver run which is too strong a consequence for a chemistry mismatch that needs iterative investigation.

### Pattern: Message Template Extensions

The [RANKING-COMPLETE] message template in solution-analyst currently contains these fields:
```
[RANKING-COMPLETE] Iteration N
Solutions: N total converted and ranked
Top solution: Rank #1: <name if known> — SMILES: <smiles>, MAE: N ppm, Matched: N/N
Strained rings: None / found in solutions <list>
Chemical plausibility: <PLAUSIBLE/QUESTIONABLE/IMPLAUSIBLE summary>
Quality: <Excellent/Good/Fair/Marginal/Poor>
Recommendation: <converge (stop) / continue (more HMBC) / escalate (problem detected)>
```

Add one new field `Aromatic warning:` after `Strained rings:`:
```
Aromatic warning: None / WARNING: N solutions lack aromatic ring despite N shifts in 110-160 ppm — likely 4J HMBC; HMBC correlations removed recommended
```

This field matches the structured-parsing pattern the orchestrator uses for loop detection. The coordinator writes CASE-PROGRESS.md based on these messages, so the field must be unambiguous.

### Pattern: Workflow Step Integration

The solution-analyst workflow section (steps 1-10) currently does not mention reading `warnings`. The workflow describes the sequence of actions the agent follows. Step 3 currently says "Run ranking: `lucy lsd rank <solutions.smi> --shifts "<shifts>"`". Steps 4-6 handle prediction and plausibility assessment but don't mention the `warnings` field.

The correct integration point is between step 3 (run ranking) and step 5 (assess chemical plausibility): after running ranking with `--format json`, parse the JSON output, extract `warnings` and check each solution's `has_aromatic_ring`. This feeds directly into Check 6 of the chemical plausibility assessment (step 5).

## Common Pitfalls

### Pitfall 1: Over-triggering the aromatic warning

**What goes wrong:** The 110-160 ppm window includes not just aromatic carbons but also alkenes (C=C sp2). A compound with 4 olefinic carbons but no aromatic ring will trigger `aromatic_shift_count >= 4` and produce a spurious warning.

**Why it happens:** The ranker uses a simple ppm window; it cannot distinguish aromatic from alkenic sp2 carbons from the shift alone.

**How to avoid:** The solution-analyst Check 6 and nmr-chemist aromatic expectation field should both note this caveat: "Aromatic or alkene sp2 carbons at 110-160 ppm." When DEPT shows CH2 at 120-130 ppm (vinyl CH2), that is an alkene not an aromatic. The check should be interpreted in context of the full spectral evidence (multiplicity, DEPT pattern).

**Warning signs:** If nmr-chemist reports CH2 carbons in the 110-160 ppm region, the "aromatic ring expected" classification may be wrong. The CONTEXT.md phrase "classic aromatic pattern" (5 sp2 CH/Cq in 127-141 ppm) is the typical positive case; pure alkenes are less common at those shift values for natural products.

**Implication for agent wording:** The new field in nmr-chemist should say "Aromatic or alkene sp2 expectation" rather than asserting "aromatic ring expected" when there is DEPT evidence of alkene CH2.

### Pitfall 2: Devils-advocate lacks ranking output to check

**What goes wrong:** The devils-advocate validates LSD files BEFORE the solver runs and solutions are ranked. By the time solutions are ranked and `has_aromatic_ring` is available, the validation step for that iteration is already done.

**Why it happens:** The workflow is: build LSD -> validate (devils-advocate) -> run LSD -> convert solutions -> rank (solution-analyst). The aromatic check on solutions is post-ranking, which is after validation.

**How to avoid:** The devils-advocate aromatic check must be framed as a post-ranking observation, not a pre-run validation. The agent can only check aromatic expectation based on nmr-chemist's [SETUP-COMPLETE] message content, not on actual solution SMILES. Its check is therefore: "NMR-chemist flagged aromatic expectation + if ranking results are available (from a previous iteration's ranking), verify warning". This is defense-in-depth, not a blocking gate.

**Practical implication:** The devils-advocate Section 3 check should be conditional: "If ranking results are available for this compound..." so it doesn't erroneously block iteration 1 validation (no ranking has happened yet).

### Pitfall 3: [RANKING-COMPLETE] field parsing ambiguity

**What goes wrong:** If the new `Aromatic warning:` field uses free text that the orchestrator's loop-detection parser doesn't expect, the coordinator may fail to detect escalation patterns correctly.

**Why it happens:** The orchestrator (case.md) parses [RANKING-COMPLETE] messages for `Recommendation:` field values to detect convergence vs continuation. The new field should not conflict with existing parsing.

**How to avoid:** Keep the new field name (`Aromatic warning:`) distinct from existing fields and place it after `Strained rings:` to maintain the existing field order for backward compatibility. The orchestrator currently parses `Recommendation:` for loop detection — the `Aromatic warning:` field is additional data the coordinator writes to CASE-PROGRESS.md, not a field the orchestrator logic parses programmatically.

### Pitfall 4: Instruction to "flag as IMPLAUSIBLE" is too strong

**What goes wrong:** If Check 6 flags ALL solutions as IMPLAUSIBLE when an aromatic warning fires, and all solutions are non-aromatic (as was the case in v4.0 ibuprofen UAT), the solution-analyst report would have NO plausible structures to recommend — leading to a confusing output with no recommendation.

**Why it happens:** "IMPLAUSIBLE" in the current framework means multiple checks fail and the structure is "likely wrong." But when all solutions are non-aromatic, the issue is with the LSD constraints, not with the individual structures per se.

**How to avoid:** Use QUESTIONABLE (not IMPLAUSIBLE) for solutions lacking `has_aromatic_ring` when an aromatic warning fires. QUESTIONABLE means "one check raises concern, needs manual review" — which is precisely the situation. IMPLAUSIBLE should be reserved for structures with multiple independent failures (strained rings + wrong functional groups, etc.).

## Architecture Patterns (Anti-Patterns to Avoid)

**Anti-pattern: Restructuring existing sections.** The agent files have a stable, tested layout. Any restructuring risks breaking the structured-message parsing patterns the orchestrator relies on (e.g., reordering fields in [RANKING-COMPLETE] would break manual CASE-PROGRESS.md authoring conventions). All edits MUST be strictly additive.

**Anti-pattern: Adding Python/CLI logic for aromatic detection to agents.** The ranking code already provides `has_aromatic_ring` in the JSON output. Agents should NOT try to independently verify aromaticity by running `lucy predict c13` and analyzing predictions. Use the signal already emitted.

**Anti-pattern: Placing the aromatic check in the workflow step numbering without updating the step count.** The solution-analyst workflow has 10 numbered steps. If a new sub-action is added between steps 3 and 4, it should be phrased as a note within step 3 or as a new unnumbered action, not as "Step 3.5" which would break the numbered sequence that agents follow.

**Anti-pattern: Making the devils-advocate check block on aromatic mismatch.** This would cause the iteration to fail whenever a compound has >= 4 sp2 carbons (including many non-aromatic natural products with conjugated systems). The check is WARNING only.

   - What we know: the ranker uses 110-160 ppm window which includes alkenes; DEPT-135 can help distinguish (aromatic CH is positive, alkene CH is positive too, but alkene CH2 is negative)
   - What's unclear: should the nmr-chemist field be phrased as "aromatic/alkene sp2 expectation" to avoid misleading solution-analyst?
   - Recommendation: Use conditional language in the field — "Aromatic ring expected" only when DEPT shows no CH2 in the 110-160 region; otherwise "sp2 carbons in 110-160 ppm (aromatic or alkene, multiplicity context needed)"

2. **Whether solution-analyst should read ranking JSON to file or from stdout**
   - What we know: workflow step 3 says "Run ranking: `lucy lsd rank ... --shifts ...`" — currently no `--format json` is specified in the workflow instruction (though the domain knowledge section shows `--format json` as a flag)
   - What's unclear: the current workflow may produce text output and the agent parses it implicitly
   - Recommendation: The workflow step should explicitly say `--format json` and pipe output to a temp file or variable, making JSON parsing reliable. Confirm by checking whether `--format json` is already in the existing workflow step.
