## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| blind-run artifacts → bookkeeper | merged.smi and LSD files produced by the fresh agent instance; bookkeeper reads them as untrusted input |
| CASE-PROGRESS.md → bypass counter | text file written by the fresh agent; bookkeeper parses it for intervention evidence |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-76B-01 | Spoofing | Agent self-report in CASE-PROGRESS.md | mitigate | Verification does not trust agent annotation of solutions. verify_case_solution.py re-runs RDKit on the actual SMILES. Agent's claimed rank/structure is not used in PASS/FAIL determination. |
| T-76B-02 | Tampering | merged.smi content (agent could write any SMILES) | mitigate | verify_case_solution.py validates SMILES via RDKit MolFromSmiles and checks formula via CalcMolFormula. An agent that fabricates a SMILES string will still fail the formula or aromatic check if the structure is wrong. |
| T-76B-03 | Repudiation | CASE-PROGRESS.md may not log all interventions | mitigate | Bookkeeper reads the full CASE-PROGRESS.md manually in Task 3. The bypass counter is a best-effort check; the mechanistic LSD-file check (SKEY/SYME/DEFF NOT absence) provides structural evidence independent of the log. |
| T-76B-04 | Denial of Service | Very large merged.smi or LSD file | accept | Local developer tool; DoS out of scope. verify_case_solution.py only reads top-3 lines effectively. |
| T-76B-SC | Tampering | No new package installs in this plan | accept | No new packages introduced. RDKit already in requirements. |
</threat_model>
